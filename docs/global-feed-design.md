# 전역 공유 피드 설계 & 동시성 학습 문서

> **목적**
> "여러 사용자가 올린 미션 인증 사진을 하나의 공유 피드에서 함께 본다"는 기능을 가정하고,
> 이때 **캐시가 어떤 역할을 하는지**, 그리고 **어떤 동시성 문제(race condition)가 터지는지**를
> 재현 시나리오와 함께 정리한 **학습 + 기획 문서**다.
>
> 현재 운영 코드(`user:{userId}:recent-records`, 유저별 캐시)와 달리,
> 이 문서는 **의도적으로 "전역 공유 키"** 를 가정해 문제를 극대화한다.

---

## 0. 한눈에 보기

| 구분 | 현재 (유저별) | 이 문서 (전역 공유) |
|------|--------------|---------------------|
| 캐시 키 | `user:{userId}:recent-records` | `feed:recent:global` (하나) |
| 쓰는 주체 | 나 자신 | **모든 유저** |
| 읽는 주체 | 나 자신 | **모든 유저** |
| race 빈도 | 매우 드묾 | **흔함** ⚠️ |
| 핵심 난제 | dual-write(삭제 실패) | dual-write + 재적재 race + stampede + hot key |

> 핵심 통찰: **race의 빈도는 "한 캐시 키를 동시에 만지는 주체 수"에 비례한다.**
> 유저별 키는 동시 주체가 "나 혼자"라 드물지만, 전역 키는 "모든 유저"라 흔하다.

## 0-1. 확정 결정 (Decision Log)

> 아래는 기획 협의로 **확정된 사항**이다. 이후 모든 설계는 이 전제를 따른다.

| # | 항목 | 결정 | 이유 / 영향 |
|---|------|------|------------|
| D1 | **피드 형태** | **전역 단일 피드** (`feed:recent:global` 키 하나) | 캐시 효율 최고(1개만 관리). 단, **hot key·stampede가 극대화**되므로 R3/R4 대응이 필수가 됨 |
| D2 | **반영 지연 허용** | **수 초** (거의 실시간) | TTL을 짧게(~60초) + **write-through**로 간다. 영구 누락은 불가(F4) |
| D3 | **무효화/갱신 방식** | **Write-through + 짧은 TTL(자가 치유)** | 완료 시 캐시에 직접 push. 실패해도 TTL로 보정. Outbox는 **선택(Phase 3)** |
| D4 | **무효화 시점** | **AFTER_COMMIT** | 재적재 race(R1) 방지 — 새 데이터가 보인 뒤 갱신 |

### 확정에 따라 "반드시" 들어가는 장치
전역 단일 키(D1) + 수 초 실시간(D2)를 동시에 만족하려면 다음이 **선택이 아니라 필수**다:

- ✅ **single-flight** (R3 stampede) — 전역 키 하나라 만료 1회 = 전체 동시 MISS
- ✅ **L1 로컬 캐시** (R4 hot key) — 전역 키 하나에 전 트래픽 집중
- ✅ **원자적 ZADD** (R5 lost update) — 동시 완료가 흔함
- ✅ **짧은 TTL + 재시도** (R2 dual-write 완화)

## 1. 기능 기획 (What)

### 1-1. 사용자 시나리오
- 홈 화면 상단에 **"방금 막 인증한 사람들"** 피드가 뜬다.
- 누구든 미션을 완료하면, 그 사진이 **전체 피드 최상단**에 거의 실시간으로 노출된다.
- 피드는 **최근 N개**(예: 20개)만 보여준다.

### 1-2. 요구사항
| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| F1 | 미션 완료 시 사진이 전역 피드에 반영 | 필수 |
| F2 | 피드는 최신순 최근 20개 | 필수 |
| F3 | 피드 조회는 빨라야 함 (DB 직격 X) | 필수 |
| F4 | 약간의 지연(수 초)은 허용, 단 **영구 누락은 불가** | 필수 |
| F5 | 트래픽 급증(이벤트 종료 직후 등)에도 버팀 | 중요 |

### 1-3. 비기능 목표
- 피드 조회 p99 latency < 50ms
- 미션 완료 → 피드 반영 지연 < 3초
- DB 부하: 피드 조회가 DB로 새는 비율 최소화

## 2. 캐시는 여기서 무슨 용도인가 (Why Cache)

### 2-1. 캐시를 쓰는 이유
피드 조회는 **읽기가 압도적으로 많다**(read-heavy). 매 조회마다 DB에서
"전체 유저 중 최근 사진 20개 정렬"을 하면:

- 정렬/조인 비용이 큼 → 느림
- 동시 조회가 몰리면 **DB가 병목** → 서비스 전체가 느려짐

그래서 **자주 안 변하지만 자주 읽히는 결과(최근 20개)** 를 캐시에 미리 만들어두고,
**대부분의 조회를 Redis(메모리)에서 처리** → DB를 보호하고 응답을 빠르게 한다.

> 캐시의 본질: **"비싼 계산 결과를, 빠른 저장소에 잠깐 복사해두고 재사용"**

### 2-2. 캐시의 3가지 역할
| 역할 | 설명 |
|------|------|
| **성능** | DB 대신 메모리에서 응답 → 빠름 |
| **DB 보호** | 읽기 트래픽을 Redis가 흡수 → DB 부하 감소 |
| **확장성** | 조회가 늘어도 DB를 안 건드림 → 수평 확장 쉬움 |

### 2-3. 캐시의 숙명적 단점 (이 문서의 주제)
캐시는 **"원본(DB)의 복사본"** 이다. 원본이 바뀌면 복사본은 **낡은다(stale).**
> **"캐시 무효화는 컴퓨터 과학의 2대 난제 중 하나다."**
> 이 문서는 그 난제가 전역 공유 환경에서 어떻게 터지는지를 다룬다.

## 3. 캐시 설계 (How)

### 3-1. 자료구조
```
Key:   feed:recent:global         (Redis List 또는 Sorted Set)
Value: [photo20, photo19, ..., photo1]   (최신순, 최대 20개)
TTL:   짧게 (예: 60초)  ← 자가 치유용
```

- **List**: push + trim으로 간단. 정렬은 "삽입 순서 = 시간 순서" 가정.
- **Sorted Set(ZSET)**: score = timestamp. 정렬/범위 조회가 정교. (권장)

### 3-2. 읽기 경로 (Cache-aside / lazy loading)
```
조회 요청
  → 서비스가 Redis 캐시 확인
     ├─ HIT  → 그대로 반환 (빠름)
     └─ MISS → 서비스가 DB 조회 → 결과를 캐시에 채움 → 반환
```
> **Cache-aside**다(read-through 아님). MISS 처리(DB 조회·재적재)를 **캐시 계층이 아니라 애플리케이션(서비스)이** 직접 한다.
> read-through라면 캐시 레포가 내부에서 DB를 로딩해 호출부엔 MISS가 안 보여야 한다.
> 따라서 이 피드의 전체 패턴 = **Cache-aside 읽기 + Write-through 쓰기 하이브리드**.

### 3-3. 쓰기 경로 (무효화 전략 선택지)
| 전략 | 동작 | 장단점 | 채택 |
|------|------|--------|------|
| A. Invalidate(삭제) | 완료 시 캐시 통째 삭제, 다음 읽기가 재적재 | 단순. but 재적재 race + stampede 위험 | ✗ |
| B. Write-through(직접 갱신) | 완료 시 새 사진을 캐시 맨 앞에 push + trim | 재적재 불필요. but 쓰기 실패 시 누락 | △ |
| **C. 하이브리드** | **write-through + 짧은 TTL로 보정** | 실무 권장 | **✅ 채택 (D3)** |

> **확정(D3): C 하이브리드.** 완료 시 `ZADD`로 캐시에 직접 push(write-through)하되,
> 쓰기가 실패해도 **TTL ~60초**가 지나면 다음 읽기가 DB에서 재적재해 자가 치유한다.
> "삭제 후 재적재"를 안 하므로 R1(재적재 race)도 구조적으로 줄어든다.

### 3-4. Write-through의 트레이드오프 (공짜가 아니다)

write-through는 stampede에 강하지만 **대가가 있다.** 무작정 좋은 게 아니라 "읽기 >> 쓰기"일 때만 이득.

| 트레이드오프 | Cache-aside (삭제) | Write-through (갱신) |
|---|---|---|
| ① 쓰기 비용 | 낮음 (지우기만) | 약간 높음 (매 쓰기마다 캐시도 갱신) |
| ② 안 읽힐 데이터 | 낭비 없음 (읽힐 때만 lazy 적재) | **낭비 가능** (아무도 안 읽어도 캐시에 씀) |
| ③ 쓰기 실패 안전성 | 안전 (지우니 다음 읽기가 DB행) | **stale 위험** (갱신 실패 시 옛 값 잔존 = dual-write) |
| ④ 코드 복잡도 | 낮음 (`delete` 한 줄) | **높음** (쓰기 경로에 "캐시 구성 로직" 필요 → 읽기/쓰기 두 곳에 흩어짐) |
| ⑤ 동시 쓰기 | 고민 없음 (그냥 삭제) | **원자 연산 필수** (`ZADD`. 통째 덮어쓰면 lost update) |
| 공유 핫키 stampede | 위험 (쓰기마다 전체 MISS) | **안전** ✅ |
| 읽기 직후 | MISS → DB | HIT ✅ |

**거래의 본질:**
```
write-through = "쓰기를 더 복잡·비싸게" 만드는 대신 "읽기를 빠르고 안전하게(stampede 방어)"
  → 읽기 >> 쓰기인 곳(전역 피드)에서만 이득
  → 쓰기 많고 읽기 적은 곳에선 손해 (②④가 그대로 비용으로)
```

> 실무 체감: ①(쓰기 부하)은 Redis 쓰기가 빨라 생각보다 작다.
> 더 아픈 건 보통 **②(낭비) + ④(복잡도) + ③(실패 시 stale)** 이다.

## 4. 동시성 문제 카탈로그 (Race Catalog)

전역 공유 피드에서 터지는 대표 문제 5종. 각각 **재현 시나리오 + 해결책**을 5장에서 다룬다.

| # | 문제 | 한 줄 정의 | 심각도 |
|---|------|-----------|--------|
| R1 | **재적재 race** | 커밋 전 무효화 → 옛 데이터로 캐시 재적재 → 영구 stale | 높음 |
| R2 | **dual-write** | DB 커밋 성공 + 캐시 쓰기 실패 → 불일치 | 높음 |
| R3 | **cache stampede** | 인기 키 만료 순간, 수백 요청이 동시에 DB로 쇄도 | 높음 |
| R4 | **hot key** | 한 키에 트래픽 집중 → Redis 단일 노드 과부하 | 중간 |
| R5 | **lost update** | 동시 갱신이 서로 덮어씀 (카운터/리스트) | 중간 |

---

## 5. Race 재현 시나리오 & 해결책

각 시나리오는 **타임라인 + "어떻게 직접 재현하나"** 를 포함한다.

---

### R1. 재적재 race (cache-aside의 함정)

#### 무엇이 문제인가
캐시 MISS 시 "DB에서 읽어 캐시에 다시 채우는(재적재)" 과정과, 무효화가 엇갈리면
**옛 데이터가 캐시에 박혀** 무효화 기회를 놓치고 **TTL까지 stale**이 된다.
이 race는 **두 가지 변종**이 있고, 둘은 해결책이 다르다.

#### 등장인물 (공통)
- **시작 DB 상태**: 사진1, 사진2, 사진3 (3개 존재)
- **스레드 A = 민수**: 지금 막 미션 완료해서 **사진4**를 올리는 중
- **스레드 B = 영희**: 그 순간 홈에서 피드를 조회하는 중

---

#### 변종 ① — 커밋 "전"에 무효화 (→ AFTER_COMMIT으로 해결 ✅)

```
시작: 캐시 = [사진3, 사진2, 사진1]   DB = [사진1, 사진2, 사진3]

  민수 (A, 미션 완료)                     영희 (B, 피드 조회)
─────────────────────────────────────────────────────────────
T1  INSERT 사진4 (아직 커밋 X)
    → 트랜잭션 안에만 존재

T2  캐시 삭제 → 캐시 = []
                                          T3  피드 조회 → 캐시 [] → MISS
                                          T4  DB SELECT
                                              → 민수 미커밋이라 사진4 안 보임
                                              → [사진3, 사진2, 사진1] 읽음
                                          T5  캐시에 써넣음
                                              → 캐시 = [사진3, 사진2, 사진1] 😱
T6  COMMIT (이제야 사진4 확정)
─────────────────────────────────────────────────────────────
결과: 캐시에 사진4 없음. 민수의 무효화(T2)는 이미 끝남.
      → 이후 모두가 [사진3,2,1]만 봄 → TTL까지 stale
```

- **원인**: 무효화(T2)를 **커밋(T6)보다 먼저** 했다.
- **해결**: 무효화를 **AFTER_COMMIT**으로 옮기면 T2가 T6 뒤로 가서 이 그림이 깨진다. ✅

---

#### 변종 ② — 옛 값을 "삭제 후"에 적재 (→ AFTER_COMMIT으로도 안 잡힘 ❌)

```
시작: 캐시 = [] (방금 TTL 만료로 비어있음)   DB = [사진1, 사진2, 사진3]

  민수 (A, 미션 완료)                     영희 (B, 피드 조회 — 느림)
─────────────────────────────────────────────────────────────
                                          T1  피드 조회 → 캐시 [] → MISS
                                          T2  DB SELECT
                                              → 민수 아직 시작 전
                                              → [사진3, 사진2, 사진1] 읽음
                                              (이 값을 쥔 채 잠깐 느려짐
                                               — GC/네트워크 지연 등)
T3  INSERT 사진4
T4  COMMIT → DB = [사진1,2,3,4]
T5  캐시 무효화(삭제) ← AFTER_COMMIT!
    → 캐시 = [] (이미 비어 사실상 헛삭제)
                                          T6  T2에서 읽은 값을 캐시에 써넣음
                                              → 캐시 = [사진3, 사진2, 사진1] 😱
─────────────────────────────────────────────────────────────
결과: 민수가 "커밋 후에 제대로" 무효화(T5)했는데도
      영희의 늦은 쓰기(T6)가 옛 값을 덮어씀 → TTL까지 stale
```

**핵심 엇갈림:**

| | 영희(B) | 민수(A) |
|---|---|---|
| DB 읽기 | T2 (커밋 **전**) → 옛 값 | |
| 캐시 무효화 | | T5 |
| 캐시 쓰기 | T6 (무효화 **후**) → 옛 값 박음 | |

영희의 **읽기는 민수 커밋보다 먼저(T2 < T4)**, 영희의 **쓰기는 민수 삭제보다 나중(T6 > T5)**.
이 "읽기 먼저 / 쓰기 나중"이 겹치면 **무효화를 커밋 후로 옮겨도** 영희가 삭제 뒤에 옛 값을 다시 깔아버려 못 막는다.

---

#### 두 변종 요약
```
변종 ①: 무효화(삭제)가 커밋보다 빠름     → AFTER_COMMIT으로 해결 ✅
변종 ②: 옛 읽기를 삭제 이후에 적재        → AFTER_COMMIT으론 못 막음 ❌
                                          → 짧은 TTL(자가 치유) + single-flight로 완화
```

#### 전역에서 왜 흔한가
유저별이면 "같은 유저가 동시에 완료+조회"라는 희귀 조건이 필요하지만,
전역 키는 **아무 유저(민수)의 쓰기 × 아무 유저(영희)의 읽기**가 겹치면 발생 → 빈번.

#### 재현 방법
- **변종 ①**: 무효화를 일부러 `@TransactionalEventListener(BEFORE_COMMIT)`(또는 트랜잭션 내부)로 옮기고, 커밋 직전 `Thread.sleep(500)`으로 창을 벌린 뒤, 그 사이 다른 스레드로 피드 조회 → 커밋 후 조회 시 새 사진이 안 보이면 재현.
- **변종 ②**: 조회 스레드의 "DB 읽기"와 "캐시 적재" 사이에 `Thread.sleep(500)`을 넣고, 그 사이에 완료 스레드가 커밋+무효화를 끝내게 타이밍을 맞춘다 → 조회 스레드의 늦은 적재가 옛 값을 덮으면 재현.

#### 해결책

먼저 **변종 ①**은 간단하다 — **무효화를 AFTER_COMMIT으로** 옮기면 끝(커밋 후 처리해야 재적재돼도 최신 보장).
어려운 건 **변종 ②**(영희가 옛 값을 늦게 박는 것)이고, 이건 아래 3가지로 다룬다.
**어느 하나도 100% 완벽하지 않아서, "크게 줄이는 방법 + 남은 걸 치유하는 방법"을 겹쳐서 깐다.**

##### (1) 짧은 TTL — "막는 게 아니라, 빨리 낫게"
stale을 예방하는 게 아니라 **오래 안 가게 시간 제한**을 거는 안전망.
```
영희가 캐시에 [사진3,2,1] 박음 (사진4 없음) ← stale 발생
  → TTL 60초면 최대 60초 뒤 캐시 자동 삭제
  → 다음 조회 MISS → DB에서 [사진1,2,3,4] 읽어 재적재
  → 사진4 보임 ✅ (최대 60초 늦게)
```
- 효과: stale이 생겨도 **최대 60초**만 지속 (현재 운영 코드의 TTL 15일과 극명히 대비).
- 한계: 그 60초간 사진4 누락 가능. but D2(수 초~수십 초 지연 허용)로 수용 가능.
- 비용: TTL이 짧을수록 캐시가 자주 비어 DB 조회 증가 → 무한정 짧게는 불가.

##### (2) 지연된 이중 삭제 — "영희가 박은 걸 한 번 더 청소"
변종 ②는 "민수가 삭제한 뒤에 영희가 옛 값을 박는" 것이므로, **민수가 잠깐 뒤 한 번 더 삭제**해 청소한다.
```
민수: 사진4 커밋
민수: 캐시 삭제 (1차)         ← 아직 영희가 안 박음
영희: 옛 값 [사진3,2,1] 박음 😱
민수: 1초 뒤 캐시 삭제 (2차)  ← 영희가 박은 옛 값을 지움 ✅
다음 조회: MISS → DB에서 [사진1,2,3,4] → 사진4 보임 ✅
```
- "1초 기다림"의 의미: 영희의 늦은 쓰기는 보통 읽기~쓰기 사이 짧은 지연에 일어나므로, **영희가 박을 시간을 준 뒤** 지운다.
- 한계: "1초"는 추측값 → 영희가 더 느리면 또 샐 수 있음 → **짧은 TTL과 병행**.

##### (3) Write-through + 원자적 추가 — "빈 창을 안 만들고, 통째 덮어쓰기 금지" (D3 채택)
**(a) 삭제 대신 추가라서 빈 창이 없음**
```
[삭제 방식] 민수: 캐시 삭제 → [] (빈 창) → 영희가 옛 값으로 채움 😱
[추가 방식] 민수: 사진4 추가 → [사진4,3,2,1] (빈 적 없음) ✅
```
삭제를 안 하니 "비어있는 틈에 옛 값이 채워지는" 변종 ①이 구조적으로 사라진다.

**(b) "통째로 덮어쓰기"를 하면 안 되는 이유 (← 핵심)**
영희가 캐시 MISS 때 **리스트 전체를 자기가 읽은 옛 스냅샷으로 통째 덮어쓰면**, 민수가 추가한 사진4도 날아간다:
```
민수: 캐시에 사진4 추가 → [사진4, 사진3, 사진2, 사진1]
영희: 옛 스냅샷으로 통째 덮어쓰기 → [사진3, 사진2, 사진1]  ← 사진4 사라짐 😱
```
이게 **"잃어버린 갱신(lost update)"** — 두 명이 같은 캐시를 동시에 만지는데, 한 명이 *읽은 걸 통째로 다시 쓰면* 그 사이 추가된 게 묻힌다.

→ 그래서 **"통째 덮어쓰기"가 아니라 "한 칸만 끼워넣기(원자적 추가)"** 를 쓴다:
```
민수: ZADD 사진4   ← "사진4 한 개만" 추가 (나머지는 안 건드림)
영희: ZADD 사진들  ← 영희도 끼워넣기만 하면 사진4를 안 지움
   → 캐시 = [사진4, 사진3, 사진2, 사진1] ✅ 둘 다 살아남음
```
`ZADD`는 Redis가 "한 원소만 끼워넣는" **단일 원자 명령**이라, 중간에 끼어들어도 서로 안 지운다.
핵심 원칙: **"읽어서 → 자바에서 합치고 → 통째로 다시 쓰기"를 하지 마라.**

##### 정리
| 해결책 | 하는 일 | 변종 ② 막나? |
|--------|---------|--------------|
| 짧은 TTL | stale이 나도 수 초 뒤 자동 치유 | 완전 차단 X, **시간 제한** (최종 안전망) |
| 지연된 이중 삭제 | 영희가 박은 옛 값을 한 번 더 청소 | 직접 겨냥, but 타이밍 추측 |
| Write-through + 원자적 추가 | 빈 창 제거 + 통째 덮어쓰기 금지 | 구조적으로 크게 줄임 |

> **우리 조합(D3): Write-through(원자적 추가) + 짧은 TTL.**
> write-through로 빈 창·통째 덮어쓰기를 없애 race를 **크게 줄이고**, 남는 미세 stale은 **짧은 TTL이 수 초 내 치유**.

### R2. Dual-write (DB ↔ Redis 불일치)

#### 무엇이 문제인가
`DB 커밋`과 `Redis 쓰기`는 **다른 시스템**이라 한 트랜잭션으로 못 묶는다.
한쪽만 성공하면 불일치.

#### 타임라인
```
① DB 커밋 성공 (사진 저장됨, 되돌리기 불가)
② Redis 쓰기 실패 (네트워크 끊김 등)
→ DB엔 있고 캐시엔 없음 → 피드에 안 뜸 → TTL까지 stale
```

#### 재현 방법
1. Redis 쓰기 직전에 `RedisTemplate`을 mock 처리해 예외를 던지게 한다.
   (또는 로컬 Redis를 `docker stop`으로 잠깐 내린다.)
2. 미션 완료 → DB엔 저장되지만 캐시 갱신 실패.
3. 피드 조회 → 새 사진 누락 확인.

#### 해결책 (완화 중심 — 원천 봉쇄는 불가)
| 방법 | 효과 | 비고 |
|------|------|------|
| **짧은 TTL** | stale 창을 TTL 길이로 한정 (예: 60초) | 가장 싸고 효과 큼 |
| **재시도 + 백오프** | transient 실패 대부분 제거 | 예외를 삼키지 말 것 |
| **Outbox 패턴** | "무효화 의도"를 DB 트랜잭션에 같이 기록 → 폴러가 재시도 → 사실상 보장 | 인프라 무거움. 이미 프로젝트에 존재 |

> **Outbox가 왜 안전한가:** 무효화 작업을 별도 Redis 호출이 아니라
> **DB INSERT(같은 트랜잭션)** 로 기록하므로, "할 일 기록"만큼은 커밋과 원자적이다.
> 이후 폴러가 그 기록을 보고 성공할 때까지 재시도 → 유실 없음.

### R3. Cache Stampede (캐시 쇄도 / thundering herd)

#### 무엇이 문제인가
인기 키 `feed:recent:global` **하나**가 만료되는 순간,
그 키를 기다리던 **수백 개 조회 요청이 동시에 캐시 MISS** → 전부 DB로 직행 → **DB 순간 과부하**.

#### 타임라인
```
T0: feed:recent:global TTL 만료
T0: 요청 300개가 동시에 도착
    → 300개 전부 캐시 MISS
    → 300개 전부 같은 무거운 DB 쿼리 실행 😱
    → DB CPU 폭증, 응답 지연, 연쇄 장애 가능
```

#### 전역 키 특유의 문제
유저별 키는 한 유저 트래픽만 몰리지만, 전역 키는 **전체 트래픽이 키 하나에 집중** → 만료 1회가 곧 전체 동시 MISS.

#### 재현 방법
1. TTL을 짧게(예: 5초) 두고, 부하 테스트 도구(k6, JMeter)로 피드 조회를 초당 수백 건 쏜다.
2. TTL 만료 타이밍에 DB 쿼리 수가 **스파이크**치는지 모니터링 → 재현.

#### 해결책
| 방법 | 동작 |
|------|------|
| **Single-flight (mutex/lock)** | MISS 시 **한 요청만** DB 조회해 캐시 채우고, 나머지는 잠깐 대기 후 캐시 사용 |
| **확률적 조기 갱신 (probabilistic early expiration)** | TTL 만료 전에 일부 요청이 미리 갱신 → 동시 만료 방지 |
| **백그라운드 갱신** | 캐시를 스케줄러가 주기적으로 미리 갱신 (요청과 분리) |
| **stale-while-revalidate** | 만료돼도 옛 값을 잠깐 더 쓰면서, 뒤에서 비동기 갱신 |

```text
Single-flight 의사코드:
  value = redis.get(key)
  if value != null: return value
  if redis.setnx(lock_key, 1, ttl=2s):   // 나만 락 획득
      value = db.query()
      redis.set(key, value, ttl)
      redis.del(lock_key)
      return value
  else:                                   // 남이 채우는 중
      sleep(50ms); return redis.get(key)  // 잠깐 기다렸다 캐시 사용
```

### R4. Hot Key (단일 키 과부하)

#### 무엇이 문제인가
`feed:recent:global` **키 하나에 모든 트래픽**이 몰리면, 그 키가 사는
**Redis 노드 하나**가 과부하. Redis Cluster를 써도 한 키는 한 노드에만 있어 **샤딩으로 못 나눔.**

#### 재현 방법
- Redis 클러스터에서 `redis-cli --hotkeys` 또는 노드별 명령 처리량 모니터링.
- 피드 조회 부하를 올리며 특정 노드 CPU만 치솟는지 확인.

#### 해결책
| 방법 | 동작 |
|------|------|
| **로컬 캐시(L1) 추가** | 애플리케이션 메모리에 1~2초 캐시 → Redis 호출 자체를 줄임 (Caffeine 등) |
| **키 복제/샤딩** | `feed:recent:global:{0..9}` 로 N개 복제, 읽기는 랜덤 분산 |
| **읽기 전용 레플리카** | Redis read replica로 읽기 분산 |

> 전역 피드는 **L1(로컬) + L2(Redis) 2단 캐시**가 잘 맞는다.
> 1~2초 로컬 캐시만 둬도 Redis 트래픽이 수십~수백 배 줄 수 있다.

---

### R5. Lost Update (동시 갱신 덮어쓰기)

#### 무엇이 문제인가
Write-through(전략 B)에서 두 완료가 동시에 캐시를 갱신할 때,
**"읽고 → 수정 → 쓰기"** 를 각자 하면 한쪽 갱신이 사라질 수 있다.

#### 타임라인 (List를 직접 읽어 통째로 다시 쓰는 잘못된 구현)
```
스레드A: 리스트 읽음 [p3,p2,p1]
스레드B: 리스트 읽음 [p3,p2,p1]   (A와 같은 상태)
스레드A: [pA,p3,p2] 로 덮어씀
스레드B: [pB,p3,p2] 로 덮어씀   ← A의 pA 사라짐 😱
```

#### 해결책
- ✅ **원자적 연산 사용**: `LPUSH` + `LTRIM` 또는 `ZADD`는 Redis가 단일 원자 명령으로 처리 → read-modify-write 안 함.
- ✅ 꼭 복합 연산이 필요하면 **Lua 스크립트**로 원자화, 또는 `WATCH/MULTI/EXEC`(낙관적 락).

> 교훈: **"읽어서 자바에서 합치고 다시 쓰기"를 피하고, 저장소의 원자적 명령에 맡겨라.**
> (앞서 미션 참가자 카운트도 같은 이유로 `UPDATE ... count+1` 원자 연산이 옳았다.)

## 6. 권장 설계 (종합)

전역 공유 피드라면, 위 문제들을 한 번에 잡는 조합:

```
[쓰기 경로] 미션 완료
  → DB 커밋 (사진 저장)
  → AFTER_COMMIT 리스너 (R1 방지)
       → 캐시 write-through: ZADD feed (원자적, R5 방지)
       → 실패 시 재시도 / Outbox 기록 (R2 완화)

[읽기 경로] 피드 조회
  → L1 로컬 캐시(1~2초) 확인 (R4 완화)
  → MISS → L2 Redis 확인
  → MISS → single-flight로 한 명만 DB 조회 (R3 방지)
  → 짧은 TTL(60초)로 자가 치유 (R2 완화)
```

| 문제 | 막는 장치 |
|------|----------|
| R1 재적재 | AFTER_COMMIT |
| R2 dual-write | 짧은 TTL + 재시도 + (선택) Outbox |
| R3 stampede | single-flight |
| R4 hot key | L1 로컬 캐시 + (선택) 샤딩 |
| R5 lost update | 원자적 ZADD/LPUSH |

## 8. 학습용 실험 메뉴 (직접 재현해보기)

> 각 race를 **일부러 재현 → 해결책 적용 후 사라지는지 확인**하는 게 핵심 학습.

| 실험 | 재현 트리거 | 기대 결과 |
|------|------------|----------|
| R1 | 무효화를 BEFORE_COMMIT으로 + `sleep` | 새 사진 영구 누락 → AFTER_COMMIT으로 바꾸면 해결 |
| R2 | Redis를 `docker stop` | 사진 누락 → TTL 짧게/재시도로 완화 확인 |
| R3 | 짧은 TTL + k6 부하 | TTL 만료 시 DB 쿼리 스파이크 → single-flight로 평탄화 |
| R4 | 단일 키 부하 집중 | 특정 Redis 노드 CPU 스파이크 → L1 캐시로 완화 |
| R5 | 동시 완료 + read-modify-write 구현 | 갱신 유실 → 원자 ZADD로 해결 |

### 실험 환경 팁
- `docker-compose`로 Redis 1대 + DB 1대 띄우기
- 부하 도구: **k6**(추천), JMeter, 또는 간단히 `ab`
- 동시성 재현: 테스트 코드에서 `CompletableFuture` 여러 개 + `CountDownLatch`로 타이밍 맞추기
- Redis 관찰: `redis-cli MONITOR`, `--hotkeys`, `INFO stats`

## 9. 핵심 교훈 요약

1. **race 빈도 = 한 키를 동시에 만지는 주체 수.** 유저별→드묾, 전역→흔함.
2. **무효화는 AFTER_COMMIT.** 새 데이터가 보인 뒤 지워야 재적재돼도 최신.
3. **dual-write는 못 막는다. 완화한다.** 짧은 TTL + 재시도 + (필요시) Outbox.
4. **전역 인기 키는 stampede/hot key가 따라온다.** single-flight + L1 캐시.
5. **read-modify-write를 피하고 저장소의 원자 연산에 맡겨라.**
6. **정확성 먼저, 성능은 나중.** Phase 0(캐시 없이)부터 쌓아 올린다.

## 9-1. 트러블슈팅 (문제 / 접근 / 결과) — 포폴용

### TS1. write-through 워밍과 TTL 자가 치유의 상충 → 적재 실패 사진 무기한 누락

**[문제]** 조회가 빈번한 피드의 전역 단일 키는 cache-aside 시 사진 등록마다 캐시 무효화로 반복
재적재가 발생해, 사진 등록 시 캐시 적재 후 TTL을 리셋해 워밍을 유지하도록 설계했다. 그러나 사진 등록 시
DB commit 후 캐시 적재가 실패하면 누락분은 키 만료 후 재적재로만 복구되는데, **잦은 사진 등록이 TTL을
계속 리셋해 키가 만료되지 않아 실패한 사진이 무기한 노출되지 않았다.**

**[접근]** "캐시 워밍 유지"와 "TTL 기반 자가 치유"가 상충함을 확인하고, 사진 등록 시 TTL을 리셋하지 않도록
둘을 분리했다(write-through는 워밍된 캐시만 보강, 콜드면 스킵해 부분 피드 노출 방지). 대신 키가 60초 주기로
만료되며 발생하는 동시 재적재(cache-stampede)는 분산락(Redisson single-flight)으로 차단했다.

**[결과]** 60초 주기 만료를 통해, 복구 불가였던 적재 실패 사진이 **최대 60초 내 자동 복구**되도록 개선했다.

> 구현 매핑: TTL 설정은 `GlobalFeedCacheRepository#rebuild`에서만(=재적재 시점), `#addItem`(write-through)은
> TTL 미변경 + 콜드 스킵. 재적재 stampede는 `GlobalFeedService#loadWithSingleFlight`(분산락 + 더블체크 + DB 폴백).

### TS2. (예정) 부하 테스트로 stampede 정량 증명
- k6로 조회 부하 → single-flight 제거 시 만료 순간 DB 쿼리 스파이크 측정 → 적용 후 평탄화 비교 (before/after).
