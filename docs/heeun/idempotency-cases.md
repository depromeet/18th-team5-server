# 멱등성이 필요한 케이스 분석

## 개요

API 서버에서 Admin 서버로 미션 로그를 전송할 때, **"알 수 없는 실패"** 상황에서 재시도 시 중복 기록이 발생할 수 있습니다. 이를 방지하기 위해 멱등성 키를 사용합니다.

---

## 실패 유형 분류

| 실패 유형 | HTTP 상태 | 재시도 필요 | 예시 |
|----------|----------|-----------|------|
| 확실한 실패 | 4xx | X (무의미) | 유효성 검사 실패, 인증 오류 |
| 알 수 없는 실패 | 5xx, 타임아웃, 네트워크 오류 | O | 서버 에러, 연결 끊김 |

**핵심 문제**: "알 수 없는 실패"는 Admin 서버에서 실제로 성공했는지 알 수 없음

---

## 멱등성이 필요한 3가지 케이스

### 케이스 1: 타임아웃으로 인한 불확실한 상태

```
┌─────────────┐                    ┌─────────────┐
│  API Server │                    │Admin Server │
└──────┬──────┘                    └──────┬──────┘
       │                                  │
       │  1. POST /mission-log            │
       │─────────────────────────────────►│
       │                                  │
       │                           2. 처리 완료 ✅
       │                           (DB 저장 성공)
       │                                  │
       │           3. 200 OK              │
       │◄ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
       │           ⏱️ TIMEOUT!            │
       │     (응답이 도착하기 전 타임아웃)    │
       │                                  │
       │  API: "실패했나? 재시도해야지"      │
       │                                  │
       │  4. POST /mission-log (재시도)   │
       │─────────────────────────────────►│
       │                                  │
       │               ⚠️ 멱등성 없으면     │
       │               중복 저장!          │
       │                                  │
```

**문제**: Admin은 성공했지만 API는 타임아웃으로 실패로 인식 → 재시도 시 중복

---

### 케이스 2: 응답 손실 (네트워크 끊김)

```
┌─────────────┐                    ┌─────────────┐
│  API Server │                    │Admin Server │
└──────┬──────┘                    └──────┬──────┘
       │                                  │
       │  1. POST /mission-log            │
       │─────────────────────────────────►│
       │                                  │
       │                           2. 처리 완료 ✅
       │                                  │
       │           3. 200 OK              │
       │         ╳ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
       │     🔌 네트워크 끊김!              │
       │     (응답 전달 실패)               │
       │                                  │
       │                                  │
       │  Outbox에 이벤트가 남아있음        │
       │  → Poller가 재시도                │
       │                                  │
       │  4. POST /mission-log (재시도)   │
       │─────────────────────────────────►│
       │                                  │
       │               ⚠️ 멱등성 없으면     │
       │               중복 저장!          │
       │                                  │
```

**문제**: 요청은 성공했지만 응답을 못 받음 → Outbox에 남아서 재시도 → 중복

---

### 케이스 3: 동시 전송 (Race Condition)

```
┌──────────────┐    ┌──────────────┐         ┌─────────────┐
│EventListener │    │   Poller     │         │Admin Server │
└──────┬───────┘    └──────┬───────┘         └──────┬──────┘
       │                   │                        │
       │ 1. Outbox 이벤트  │                        │
       │    발견           │                        │
       │                   │                        │
       │  2. POST 전송 ────┼───────────────────────►│
       │                   │                        │
       │                   │ 3. 같은 이벤트          │
       │                   │    아직 Outbox에 존재   │
       │                   │    (삭제 전)            │
       │                   │                        │
       │                   │  4. POST 전송 ─────────►│
       │                   │     (동시 전송!)        │
       │                   │                        │
       │         5. 200 OK │                        │
       │◄──────────────────┼────────────────────────│
       │                   │         6. 200 OK      │
       │                   │◄───────────────────────│
       │                   │                        │
       │               ⚠️ 멱등성 없으면 중복 저장!    │
```

**문제**: EventListener와 Poller가 동시에 같은 이벤트 전송 → 중복

## 해결책: 멱등성 키

### Admin Server 처리 흐름

```
     요청 수신
         │
         ▼
┌─────────────────────┐
│ 멱등성 키 추출       │  idempotencyKey = {userUuid}:{missionId}:{date}
│ (헤더 또는 페이로드)  │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐       ┌──────────────────┐
│ DB에 키가 존재?      │──Yes──►│ 200 OK 반환      │
│ (이미 처리됨?)       │       │ (처리 안 함)      │
└─────────┬───────────┘       └──────────────────┘
          │ No
          ▼
┌─────────────────────┐
│ 로그 저장           │
│ + 멱등성 키 저장     │
└─────────┬───────────┘
          │
          ▼
    200 OK 반환
```

### 멱등성 키 구조

```
{userUuid}:{missionId}:{completedDate}:{hash}
```

예: `abc123:42:2024-01-15:a1b2c3d4`

## 코드상의 멱등성 처리

### API Server (AdminClient.java)

409 Conflict를 성공으로 처리:

```java
} catch (HttpClientErrorException.Conflict e) {
    // 409 Conflict: 이미 존재하는 요청
    log.info("미션 로그 이미 존재 (409): missionId={}", payload.missionId());
    return new SendResult.Success(eventId);  // ← 성공으로 처리!
}
```

### Admin Server (StatsService.java)

멱등성 키로 중복 체크:

```java
@Transactional
public void saveMissionLog(MissionLogRequest request) {
    String idempotencyKey = generateIdempotencyKey(
        request.userUuid(),
        request.missionId(),
        request.completedAt().toLocalDate()
    );

    // 이미 존재하면 무시
    if (userMissionLogRepository.existsByIdempotencyKey(idempotencyKey)) {
        log.info("이미 존재하는 미션 로그: {}", idempotencyKey);
        return;  // 또는 409 반환
    }

    userMissionLogRepository.save(createMissionLog(request, idempotencyKey));
}
```

## 결론

| 케이스 | 원인 | 멱등성 없을 때 | 멱등성 있을 때 |
|-------|------|--------------|--------------|
| 타임아웃 | 응답 지연 | 중복 저장 | 중복 방지 ✅ |
| 응답 손실 | 네트워크 끊김 | 중복 저장 | 중복 방지 ✅ |
| 동시 전송 | Race Condition | 중복 저장 | 중복 방지 ✅ |

**멱등성 키를 사용하면 "알 수 없는 실패" 상황에서도 안전하게 재시도할 수 있습니다.**
