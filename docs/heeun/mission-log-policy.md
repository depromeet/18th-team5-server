# 미션 완료 로그 전송 정책

## 개요

API 서버에서 Admin 서버로 미션 완료 로그를 전송하는 이유와 정책을 정리한 문서입니다.

---

## 데이터 흐름

```
┌─────────────────┐         ┌─────────────────┐
│   peektime-api  │  ────►  │  peektime-admin │
│   (API 서버)     │  HTTP   │   (Admin 서버)   │
└─────────────────┘         └─────────────────┘
        │                           │
        ▼                           ▼
   ┌─────────┐                ┌─────────┐
   │ API DB  │                │ Admin DB│
   └─────────┘                └─────────┘
```

---

## 왜 API → Admin으로 데이터를 보내는가?

### 1. 랭킹 기반 보상 시스템

Admin에서 미션 수행 상위 사용자를 확인하고, **상위권 사용자에게 보상을 제공**하기 위함입니다.

| 목적 | 설명 |
|------|------|
| 랭킹 확인 | 미션 완료 횟수 상위 사용자 조회 |
| 보상 지급 | 상위권 사용자에게 리워드 제공 |

### 2. 아키텍처 분리

Admin과 API가 **별도 DB**를 사용하는 구조입니다.

```
❌ Admin이 API DB 직접 접근 → 아키텍처 위반
✅ API가 데이터를 Admin에 전송 → 올바른 설계
```

각 서버가 자신의 DB만 관리하고, 필요한 데이터는 API를 통해 전달합니다.

---

## 왜 재시도가 필요한가?

### 보상이 연결된 데이터

```
미션 완료 데이터 유실
       ↓
랭킹 순위 왜곡
       ↓
보상 받아야 할 사용자가 누락
       ↓
사용자 클레임 / 불공정 이슈
```

**보상**이 걸려있기 때문에 데이터의 정확한 전달이 필수입니다.

### 데이터 정책

| 항목 | 정책 |
|------|------|
| 데이터 유실 | 불가 |
| 전송 방식 | At-Least-Once (최소 1회 보장) |
| 중복 처리 | 멱등성 키로 방지 |
| 재시도 | Outbox 패턴 + Poller |

---

## 정리

| 질문 | 답변 |
|------|------|
| 왜 Admin으로 보내나요? | Admin에서 랭킹 현황을 직접 보고 상위권에 보상을 주기 위해 |
| 왜 API DB에서 집계 안 하나요? | Admin/API가 별도 DB이고, Admin이 API DB에 직접 접근하면 아키텍처 위반 |
| 왜 재시도가 필요한가요? | 보상 지급에 필요한 데이터라서 유실되면 안 됨 |
